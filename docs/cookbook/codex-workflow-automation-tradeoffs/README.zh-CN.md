# 我如何思考 Codex 工作流自动化

[English](./README.md)

关联项目：[mo2g/codex-composer](https://github.com/mo2g/codex-composer)

## 问题背景

我最初的目标其实很直接：把一个常规工程工作流干净地映射到 Codex 上。理想路径大概是这样：

1. 接收需求
2. 分析需求
3. 判断是否可以拆分成 A、B 两个并行任务
4. 让 A、B 独立开发
5. A、B 分别完成验证
6. A、B 分别提交 commit
7. 把 AB 集成
8. 做最终验证
9. 合并回主线

表面上看，这像是在解决一个“怎样自动化更多”的问题。但做了更多测试之后，我不再这样定义它。真正的问题是：面对具体用户和具体仓库时，什么程度的自动化还能保持可靠、可审查，而且成本可接受。

## 演进过程

### 阶段 1：从传统软件工作流出发

我的第一反应，是把标准工程协作模型直接搬到 Codex 里。如果人类开发中的 A/B 并行有价值，那么 AI 辅助执行理论上也应该能复制这种价值。

这个想法在结构上没问题，但它掩盖了一个前提：人类并行从来都不只是“同时做事”，它还依赖清晰的 ownership、review 纪律和共享上下文。Codex 并不会自动消除这些要求。

### 阶段 2：看完官方文档后，subagents 很像最自然的答案

在看过官方文档之后，subagents 看起来像是实现 A/B 并行开发最直接的手段。如果目标是“让 A 和 B 同时推进”，那么 subagents 很容易被理解成最原生、最顺手的工具。

这是最有诱惑力的阶段，因为它在纸面上非常优雅：先拆分，再分发，再汇合。

### 阶段 3：深入学习和测试后，暴露出缺陷与隐性成本

当我花更多时间做学习和测试之后，我开始不再认为 subagents 应该是默认答案。问题不只是功能层面的，更是运维层面的：

- 执行状态更难直接看清
- 失败更难被准确定位
- 上下文复制会推高 token 开销
- 集成阶段依然需要人类级别的 review

相比之下，Codex App 自带的线程更可靠，也更容易检查。thread-first 的方式没有那么“魔法”，但状态更清楚，而这在真实工作中非常重要。

### 阶段 4：真正的争论变成了“全自动化 vs 半自动化”

到了这里，真正重要的比较对象已经不再是孤立的“threads 还是 subagents”。更核心的设计问题变成：到底应该把自动化推到什么程度。

我后来把它理解成一条连续光谱：

- 手动
- 半自动
- 受监督的并行执行
- 自动化较重 / 接近全自动

这样看问题会更准确。subagents 只是更大自动化博弈中的一种实现手段，而不是“更先进”本身。

### 阶段 5：我目前的立场是按人群分层，而不是给一个通用答案

我现在并不认为所有人都应该使用同一种 Codex 工作流。更合理的答案取决于预算、对不确定性的容忍度、对可审计性的要求、仓库成熟度，以及用户想保留多少控制权。

所以我现在更倾向于按人群给建议，而不是宣布一种“最佳流程”。

## 设计决策

我目前偏好的默认方案是保守的：

- current thread first
- 只有在工作确实可以拆开时才做 split
- 优先使用 Codex App thread，加可选 worktree，再考虑 subagents
- 保留显式的 `verify` 和 `commit` gate
- merge 保持手动

我更偏向 thread-first，主要有几个现实原因：

- 执行状态更可见
- 人工 review 更容易分阶段进行
- 隐含的协调假设更少
- 不容易意外放大 token 成本
- 对大多数个人用户来说，比自动化更重的编排更合适

这并不等于说 subagents 永远没用。更准确的说法是：我不希望默认工作流建立在一个很多仓库和很多用户都还没准备好运营的协调层之上。

## 踩坑

我不会把“subagents 不好”写成绝对结论。我现在的看法更具体一些：它会带来一些很容易被低估的风险。

- 协调过程不透明，更难知道到底哪一步发生了什么
- 当失败跨越多个执行边界时，调试难度会上升
- 上下文复制容易让 token 消耗增长得比预期更快
- 表面上的并行感，会制造吞吐提升的错觉
- review 负担往往不会消失，而是被推迟到 integration 阶段

最容易犯的错误，是只统计“并行执行时间”，却不去计算 integration、诊断和 review 的成本。

## 权衡

我现在更愿意把 Codex 工作流看成一条自动化光谱，而不是一个二元开关。

### 手动

由人来主导拆分、执行顺序、验证和 merge。它更慢，但可见性最高。

### 半自动

Codex 帮助做规划、起草、实现和 review 准备，但人仍然掌控 split 点、验证、commit gate 和 merge。这是我给大多数个人用户的默认建议。

### 受监督的并行执行

当任务可以被干净拆分，而且 integration 方案已经明确时，引入并行 thread 或 worktree。这种方式可以有效，但前提是 ownership 边界必须足够清晰。

### 自动化较重 / 接近全自动

这种模式在追求速度时很有吸引力，但前提是操作者愿意接受更多不确定性、更大的成本波动，以及更重的下游 review 负担。我不认为它适合作为广泛默认值。

更高程度的自动化，只有在仓库契约、验证机制、ownership 边界、回滚纪律和 review 习惯都已经成熟时，才会持续带来收益。否则它更多是在转移复杂度，而不是消灭复杂度。

## 决策矩阵

| 用户类型 | 预算层 | 偏好流程形态 | 自动化程度 | 原因 |
| --- | --- | --- | --- | --- |
| 个人用户 | 大约 `$20/月` / 低预算 | current thread first，必要时才显式拆分 | 手动或半自动 | 通常更关注成本，需要高可见性，相比激进自动化，更受益于可预测控制 |
| 高投入用户 | `$100+/月` / 高预算 | 更愿意尝试自动化更重的流程或更强并行 | 受监督并行到自动化较重 | 当速度比可审计性和成本波动更重要时，更能接受更高投入 |
| 团队或高级操作者 | 预算可变，但依赖成熟度 | thread + worktree，在仓库契约足够强时考虑受监督并行 | 半自动到受监督并行 | 如果 review 纪律、验证能力和回滚习惯已经存在，就能吸收协调成本 |

预算之所以重要，不只是因为它是一个价格数字。它其实也是一种容忍度代理变量：对 token 消耗的容忍度、对不确定性的容忍度，以及对控制权损失的容忍度。

## 可复用经验

- 并行首先是协调问题，其次才是执行问题。
- 对大多数真实工作来说，可见状态优于隐藏编排。
- 可靠性和可审查性，比“看起来更原生”或“更全自动”更重要。
- 自动化应该由用户分层和仓库成熟度决定，而不是由新鲜感决定。
- 一个在 demo 里很优雅的流程，到了真实使用场景中，仍然可能非常昂贵地调试。

## 下一步

- 继续验证 subagents 在什么情况下值得它的成本，而不是绝对否定它。
- 继续用真实任务比较 thread-first 和 subagent-first，而不是只看抽象图示。
- 继续细化按预算分层的推荐方式。
- 在推荐更激进自动化之前，先把仓库契约做得更强。
