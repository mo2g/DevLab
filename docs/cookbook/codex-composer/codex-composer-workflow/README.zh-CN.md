# Codex Composer 开发经验

[English](README.md)

关联项目：[mo2g/codex-composer](https://github.com/mo2g/codex-composer)

## 问题背景

很多团队开始使用 Codex 时，真正起作用的并不是某一条 prompt，而是围绕仓库逐渐形成的一套隐性约定：什么时候先规划，什么时候直接实现，验证命令从哪里找，哪些事情必须人工 review，哪些知识应该长期保留在仓库里。这些约定如果只停留在聊天记录里，就很难复制，也很难让团队成员形成一致预期。

Codex Composer 想解决的核心问题，不是“怎么写一个更聪明的 prompt”，而是“怎么把 AI coding workflow 变成一个仓库级别可移植的工作方式”。

## 设计决策

- 把仓库契约前置到 `AGENTS.md`，让规则先于任务存在。
- 把验证命令收敛到 `.codex/config.toml`，避免每次都重新解释测试入口。
- 只保留少量高价值能力，如规划、实现、merge readiness，而不是堆一整套复杂协议。
- 用模板和安装脚本做 bootstrap，而不是把每个仓库都改造成一个新的平台。
- 明确保留人工 review 和人工 merge，避免把交付边界模糊给 AI。

## 踩坑

- 一开始很容易把“可复用”理解成“配置项越多越好”，结果模板本身会变得比目标仓库还复杂。
- 如果仓库契约写得过于抽象，AI 会看起来像遵守了流程，实际上却没有执行可验证的动作。
- 过度依赖单线程工作流会让协作成本升高，尤其是当任务天然可以拆分时。
- 安装模板时如果侵入 README 或现有目录结构太多，用户会立刻觉得它不是增强，而是接管。

## 权衡

- 轻量模板和强约束流程之间，我更偏向前者，因为仓库差异远比想象中大。
- 自动 merge 和人工 merge 之间，我明确选择人工 merge，因为责任边界必须清晰。
- 多技能覆盖和少技能高质量之间，我更倾向少量高质量技能，先解决高频问题。
- “告诉 AI 怎么做”与“把仓库变成 AI 能理解的环境”之间，我认为后者更可持续。

## 可复用经验

- 真正可复制的 AI workflow，应该先写仓库契约，再写任务提示。
- 所有验证命令都应该有稳定入口，不要把测试方法藏在人的记忆里。
- 模板的价值在于降低重复决策，而不是制造新的学习成本。
- 如果一个 AI 工作流不能解释人该在哪一步介入，它就不够适合真实团队。

## 下一步

- 继续优化 bootstrap 体验，让接入现有仓库时更少摩擦。
- 增加更强的示例仓库和迁移说明，降低第一次理解成本。
- 细化文档，把“为什么这样设计”表达得比“有哪些文件”更清楚。
- 逐步沉淀更多能跨仓库复用的经验，而不是急于扩成大而全框架。
