# Codex Composer 轻量标准工作流收敛

[English](README.md)

关联项目：[mo2g/codex-composer](https://github.com/mo2g/codex-composer)

## 问题背景

这次迭代的重点不是“再加一层自动化”，而是把 Codex App 的默认工作流收窄到足够轻、足够稳、足够适合日常使用。

这次要回答的核心问题是：

1. 默认路径里什么该保留
2. 什么应该变成可选项，而不是强制项
3. 什么时候先澄清意图，再开始规划
4. 当目标仓库技术栈未知时，验证应该怎么做

## 设计决策

- 默认流程保持 `current thread first`。
- `split` 和 `worktree` 继续保留，但只作为可选工具，不作为默认心理模型。
- 只保留 `planner`、`implementer`、`change-check` 三个技能。
- `planner` 先澄清意图，再输出可执行的计划。
- `change-check` 根据真实仓库和本次 diff 动态决定测试与验证方式，必要时先补直接测试。
- merge 继续保持手动，但不再额外引入 merge checklist。
- `.codex/config.toml` 只作为仓库可选配置，而不是每个目标仓库的默认真相源。

## 踩坑

- 很容易把“轻量”写在文档里，却在默认安装里保留太多协议感。
- 固定验证文件在目标仓库技术栈未知时，容易变成隐含前提。
- 如果 `planner` 跳过意图澄清，后续就会退化成普通计划生成器，而不是好的入口。
- 如果 `change-check` 只跑已有测试，就会漏掉为新行为补直接覆盖这一步。

## 权衡

- 在 source repo 默认值和 target repo 默认值之间，我现在更偏向保留 source repo 默认值，让 target config 变成可选。
- 在单独的澄清技能和更强的 planner 之间，我更偏向后者，因为技能面更小。
- 在静态 verify hooks 和动态验证之间，我更偏向动态验证，因为它更适合未知目标仓库。
- 在手动 merge 和 merge checklist 之间，我保留手动 merge，去掉 checklist，因为它增加的是仪式感，不是责任边界。

## 可复用经验

- 轻量化本质上是减法。
- 默认路径应该能用一句话讲清楚。
- 仓库契约有价值，但前提是它不能比目标仓库本身还重。
- 验证应该跟着变更走，而不是反过来。
- 一个健康的 AI workflow，应该能说清楚人从哪里开始接管判断。

## 下一步

- 继续让文档和技能保持同一套轻量叙事。
- 继续验证 `planner` 能否吸收更多意图澄清而不变重。
- 继续在真实目标仓库里压力测试动态验证。
- 只有在某个模式足够稳定时，再把它写进更多 cookbook 条目。
