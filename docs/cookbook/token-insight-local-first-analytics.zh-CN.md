# Token Insight 开发经验

[English](./token-insight-local-first-analytics.md)

关联项目：[mo2g/token-insight](https://github.com/mo2g/token-insight)

## 问题背景

AI coding tools 的使用数据天然是碎片化的。不同工具把 token、模型、来源、时间、成本相关信息散落在本地文件、缓存、日志或导出结果里。很多团队一旦想做 usage analytics，第一反应就是接入外部 SaaS，但这样会立刻遇到两个问题：一是隐私和本地数据边界，二是各家工具的数据结构根本不统一。

Token Insight 的出发点是把“本地优先”当成产品立场，而不是部署方式。只有先把本地 usage event 整理清楚，后续无论是做分析、分享还是对外同步，才有稳定基础。

## 设计决策

- 先做 ingestion 和 normalization，再做 dashboard，避免 UI 先行导致数据模型反复返工。
- 默认使用 SQLite，把单机可运行和可检查放在首位。
- 把后端采集与前端可视化拆开，让数据模型演进不被界面节奏绑死。
- 把刷新、导出、分享图等操作做成脚本，降低“只是想看一下数据”的门槛。
- 明确支持多来源输入，避免产品从第一天起就被绑定在单一工具生态里。

## 踩坑

- 各家工具的 usage artifacts 并不稳定，字段命名和时间格式很容易变化。
- 成本估算如果直接绑定实时价格，会让历史数据解释变得混乱。
- 数据一旦开始做可视化，用户对准确度的期待会迅速提高，任何归一化误差都会放大。
- 本地优先虽然提升隐私可信度，但也意味着安装、刷新、排错都必须足够简单，否则产品会失去吸引力。

## 权衡

- 本地优先与多人协作之间，我优先保留本地真实性，再考虑可选同步。
- SQLite 与更复杂的分析型存储之间，我优先选择简单、可迁移、易调试。
- 通用 schema 与来源特定字段之间，我更倾向先建立稳定主干，再逐步扩展边缘字段。
- 炫目的图表和可解释的数据模型之间，我优先后者，因为 analytics 产品的信任来自解释力。

## 可复用经验

- 如果数据来源天然混乱，先定义标准化事件模型，再决定页面长什么样。
- “本地优先”应该贯穿产品叙事、部署方式和数据边界，而不是只停留在一句口号。
- 导出能力不是附属功能，它往往决定了这个系统能不能真正进入用户工作流。
- 可分享的结果视图非常重要，因为很多分析工具真正传播出去，靠的是用户愿意截图或导出结果。

## 下一步

- 扩大适配来源覆盖，让更多 AI coding tools 的 usage data 可以进入同一视图。
- 强化 parser fixtures 和自动化验证，降低适配层演进带来的回归风险。
- 继续打磨 dashboard 表达，让趋势、分布和异常更容易被快速读懂。
- 在不破坏本地优先前提下，探索更安全的同步和分享机制。
