# AgentScan 工程思考

[English](./agentscan-agent-security-notes.md)

关联项目：[AutoScan/agentscan](https://github.com/AutoScan/agentscan)

## 问题背景

AI Agent 的部署扩散速度非常快，但很多部署者并不具备传统安全工程经验。结果就是：服务启动了、端口开放了、默认配置保留了、访问控制缺失了，而运维侧甚至不知道这些实例已经暴露。传统扫描器能告诉你“这里有个端口”，却很难继续说明“这是不是一个 AI Agent，它暴露了哪些能力，风险到底有多高”。

AgentScan 的价值在于把 AI Agent 作为一个独立资产类别来对待，而不是把它混在普通 Web 服务或通用中间件里。

## 设计决策

- 扫描流程分层为 L1、L2、L3，把发现、识别、验证拆开处理。
- 不只做资产发现，还补上任务调度、告警、报告和前端展示，保证结果能被运营。
- 后端围绕扫描管线设计，前端则围绕“风险解释”和“任务状态”来组织信息。
- 配置优先级清晰，便于从开发环境过渡到更正式的部署环境。
- 对安全检测保持平台化思路，而不是只为单一 Agent 产品写一次性脚本。

## 踩坑

- 指纹识别如果做得太激进，会造成误报，后续所有漏洞判断都会被污染。
- 漏洞验证如果没有安全边界，很容易从“检查”变成“扰动目标”。
- 一旦接入任务、告警和报告，项目复杂度会远高于单纯 CLI 扫描器。
- 安全产品的使用者往往不满足于“检测到了”，他们更需要“为什么重要”和“下一步怎么处理”。

## 权衡

- 扫描速度与识别准确率之间，必须接受不是所有目标都适合激进并发。
- 覆盖更多 Agent 类型与深入支持少量类型之间，需要持续平衡。
- CLI 体验和平台化能力之间，项目需要同时照顾自动化调用与可视化运营。
- 安全验证的深度不能只看技术上能不能做到，还要看默认行为是否足够克制。

## 可复用经验

- 如果一个新系统正在快速普及，它很可能很快就会成为新的资产面和攻击面。
- 分层扫描架构比“单次大而全探测”更容易维护，也更利于解释结果。
- 安全工具的可信度来自保守默认值、可追踪结果和清晰的 remediation 语义。
- 做 AI 安全产品时，技术检测和面向运营的表达同样重要。

## 下一步

- 增强指纹和漏洞数据库的可维护性，让规则迭代速度跟上生态变化。
- 继续补强权限、限流、观测和健康检查等平台级能力。
- 在更大规模扫描场景下优化并发和任务调度策略。
- 让报告和告警更贴近真实治理流程，而不只是扫描结果的导出物。
